# One Fact, One Source 快速指南

本指南提供人類可讀的繁體中文說明；Agent 的正式執行規則以 [`SKILL.md`](../skills/one-fact-one-source/SKILL.md) 為準。

## 一句話理解

同一項商業事實，只保留一個權威來源與一個正式名稱；畫面、API、排程或其他模組只能使用它的結果，不得各自重算。

這不是程式設計入門教材，而是提供給已經使用 Claude Code、Codex 或其他 Coding Agent 的開發者，用來避免 AI 在專案裡建立第二套商業規則。

## 適用情況

適合檢查或修改：

- 金額、門檻、期限與計算規則；
- 狀態、權限、分類與資格判斷；
- 共用常數、型別、schema 與 contract；
- 前後端或跨 repository 重複的商業邏輯；
- 被當成正式答案使用的快取、衍生欄位或 read model。

不適合因為程式外觀相似就直接合併：

- 純排版或沒有商業意義的 helper；
- 商業名稱、資料歸屬或邊界不同的規則；
- 只是在不同平台呈現同一結果的 UI 與 transport adapter。

## 常用術語

- **Canonical source（權威來源）**：這項規則唯一可以被認定為正式答案的位置。
- **Consumer（使用端）**：使用這項答案的畫面、API、排程或其他模組。
- **Projection（投影／衍生結果）**：由權威資料計算出來、方便查詢或顯示的結果，不是正式真相。
- **Drift（漂移）**：原本應一致的規則或鏡像，後來各自修改而產生差異。

## 三種工作模式

### Audit only：只讀盤點

只搜尋、比較並產出報告，不修改檔案。適合第一次接觸規則，或尚未確認哪個來源正確時使用。

### Small change：小修改

針對一項既有商業事實做局部修改。Agent 應先找到並重用權威來源，不得增加平行實作。若發現重複或衝突，先回報，不擅自擴大成重構。

### Consolidation or refactor：正式整併

在已授權的範圍內建立或確認權威來源、遷移所有已知使用端、移除舊定義，並完成測試、舊名稱與漂移檢查。

## 啟動方式

不同 Agent 的手動呼叫語法不同：

```text
Codex：       使用 $one-fact-one-source
Claude Code： /one-fact-one-source
其他 Agent：  直接要求使用 one-fact-one-source Skill
```

啟動 Skill 後，再貼上後面的只讀盤點或正式整併任務內容。

## 名稱變更與相容性

內部函式改名與公開識別碼改名的風險不同。API 欄位或路由、資料庫欄位或值、事件名稱、設定鍵、序列化名稱及跨 repository contract，未經明確授權不得改名。

若舊名稱因相容性必須保留，它只能作為轉接層並委派給權威來源；報告中必須說明保留原因。

## 三種判斷案例

### 應該合併

購物車、結帳畫面與後端 API 都用 `subtotal >= 1000` 判斷免運資格。三者代表同一項商業事實，應共同呼叫一個權威函式。

畫面可以顯示不同文字或顏色，API 也可以使用不同 response shape，但不得重新計算免運資格。

### 不應合併

`paymentDueDate = date + N days` 與 `contractEndDate = date + N days` 的程式形狀相似，但一個代表付款截止日，另一個代表租約結束日。只要商業意義、歸屬或邊界不同，就不應合併成同一個函式。

### 必須由人決定

同一方案限制在不同檔案中分別寫成「1 個專案」、「3 個專案」與「不限專案」。Agent 應列出位置、名稱、實際行為與可能版本來源，停止修改並請任務負責人決定，不得自行挑選看起來合理的答案。

## 只讀盤點提示詞

```text
盤點以下商業規則：

[請描述規則、功能或可能涉及的檔案]

本次只進行 Audit only，不修改任何檔案。

請找出並回報：
1. 所有可能的規則定義位置；
2. 每個定義的名稱、檔案與實際行為；
3. 所有已知使用端；
4. 哪些位置只是呈現或傳遞結果，哪些位置正在重新計算；
5. 各實作是否完全一致；
6. 使用的搜尋方法、涵蓋範圍與已知盲區；
7. 證據足夠時，建議的唯一權威來源與正式名稱；
8. 應遷移或移除的舊來源；
9. 無法存取的 mirror repository 或 artifact；
10. 需要由人決定的衝突。

若現有實作互相衝突，只能列出候選與差異，不得自行決定正確版本。
```

## 正式整併提示詞

```text
整併以下商業規則：

[請描述已確認的規則、權威來源及允許修改的範圍]

請在不改變既有行為的前提下：
1. 保留或建立唯一權威來源與正式名稱；
2. 記錄搜尋方法與盲區，並遷移在涵蓋範圍內找到的使用端；
3. 移除舊函式、常數、型別、狀態集合與行內判斷；
4. 未經明確授權，不得更改公開或持久化識別碼；
5. 保留並說明刻意存在的差異與相容性名稱；
6. 使用現有測試框架補測試；若專案沒有測試框架，不得擅自導入；
7. 執行專案現有的測試、型別檢查、lint 與 build；
8. 搜尋舊名稱、複製條件、magic values 與跨 repository 漂移；
9. 依完成報告格式回報結果。

若發現未授權的行為差異或規則衝突，停止並回報，不得自行選擇。
```

## 完成報告格式

```text
1. 商業事實：
2. 權威名稱與檔案：
3. 定義數量（修改前／修改後）：
4. 搜尋方法與涵蓋範圍：
5. 已知搜尋盲區：
6. 已遷移使用端：
7. 已移除的舊定義與保留的相容性名稱：
8. 刻意保留的差異：
9. 授權的行為或識別碼變更：
10. 測試、型別、lint 與 build 結果：
11. clone／舊名稱／magic value 檢查結果：
12. 跨 repository 存取與漂移結果：
13. 尚未解決的衝突或風險：
```

沒有使用某項工具或專案不存在某項檢查時，應明確寫「未使用」或「專案未提供」，不要假裝已執行。

## 可選安全限制

工程操作權限不屬於這個 Skill 的核心規則，應由全域規則、專案規則或單次工單決定。需要限制時，可在提示詞最後加入：

```text
本次不得 commit、push、建立 PR、執行 migration 或部署。
不得修改指定範圍以外的檔案；若需要擴大範圍，先停止並回報。
```
