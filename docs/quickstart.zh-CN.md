# One Fact, One Source 快速指南

本指南提供便于阅读的简体中文说明；Agent 的正式执行规则以 [`SKILL.md`](../skills/one-fact-one-source/SKILL.md) 为准。

## 一句话理解

同一项业务事实，只保留一个权威来源与一个正式名称；界面、API、定时任务或其他模块只能使用它的结果，不得各自重新计算。

这不是编程入门教材，而是提供给已经使用 Claude Code、Codex 或其他 Coding Agent 的开发者，用来避免 AI 在项目里建立第二套业务规则。

## 适用情况

适合检查或修改：

- 金额、门槛、期限与计算规则；
- 状态、权限、分类与资格判断；
- 共享常量、类型、schema 与 contract；
- 前后端或跨 repository 重复的业务逻辑；
- 被当成正式答案使用的缓存、派生字段或 read model。

不适合因为代码外观相似就直接合并：

- 纯布局或没有业务含义的 helper；
- 业务名称、数据归属或边界不同的规则；
- 只是在不同平台呈现同一结果的 UI 与 transport adapter。

## 常用术语

- **Canonical source（权威来源）**：这项规则唯一可以被认定为正式答案的位置。
- **Consumer（使用端）**：使用这项答案的界面、API、定时任务或其他模块。
- **Projection（投影／派生结果）**：由权威数据计算出来、方便查询或显示的结果，不是正式真相。
- **Drift（漂移）**：原本应一致的规则或镜像，后来各自修改而产生差异。

## 三种工作模式

### Audit only：只读盘点

只搜索、比较并产出报告，不修改文件。适合第一次接触规则，或尚未确认哪个来源正确时使用。

### Small change：小修改

针对一项现有业务事实做局部修改。Agent 应先找到并复用权威来源，不得增加平行实现。若发现重复或冲突，先报告，不擅自扩大成重构。

### Consolidation or refactor：正式整合

在已授权的范围内建立或确认权威来源、迁移所有已知使用端、移除旧定义，并完成测试、旧名称与漂移检查。

## 启动方式

不同 Agent 的手动调用语法不同：

```text
Codex：       使用 $one-fact-one-source
Claude Code： /one-fact-one-source
其他 Agent：  直接要求使用 one-fact-one-source Skill
```

启动 Skill 后，再粘贴后面的只读盘点或正式整合任务内容。

## 名称变更与兼容性

内部函数改名与公开标识符改名的风险不同。API 字段或路由、数据库字段或值、事件名称、配置键、序列化名称及跨 repository contract，未经明确授权不得改名。

若旧名称因兼容性必须保留，它只能作为转接层并委派给权威来源；报告中必须说明保留原因。

## 三种判断案例

### 应该合并

购物车、结账界面与后端 API 都使用 `subtotal >= 1000` 判断免运费资格。三者代表同一项业务事实，应共同调用一个权威函数。

界面可以显示不同文字或颜色，API 也可以使用不同 response shape，但不得重新计算免运费资格。

### 不应合并

`paymentDueDate = date + N days` 与 `contractEndDate = date + N days` 的代码形状相似，但一个代表付款截止日期，另一个代表合同结束日期。只要业务含义、归属或边界不同，就不应合并成同一个函数。

### 必须由人决定

同一方案限制在不同文件中分别写成“1 个项目”、“3 个项目”与“不限项目”。Agent 应列出位置、名称、实际行为与可能的版本来源，停止修改并请任务负责人决定，不得自行挑选看起来合理的答案。

## 只读盘点提示词

```text
盘点以下业务规则：

[请描述规则、功能或可能涉及的文件]

本次只进行 Audit only，不修改任何文件。

请找出并报告：
1. 所有可能的规则定义位置；
2. 每个定义的名称、文件与实际行为；
3. 所有已知使用端；
4. 哪些位置只是呈现或传递结果，哪些位置正在重新计算；
5. 各实现是否完全一致；
6. 使用的搜索方法、覆盖范围与已知盲区；
7. 证据充足时，建议的唯一权威来源与正式名称；
8. 应迁移或移除的旧来源；
9. 无法访问的 mirror repository 或 artifact；
10. 需要由人决定的冲突。

若现有实现互相冲突，只能列出候选与差异，不得自行决定正确版本。
```

## 正式整合提示词

```text
整合以下业务规则：

[请描述已确认的规则、权威来源及允许修改的范围]

请在不改变现有行为的前提下：
1. 保留或建立唯一权威来源与正式名称；
2. 记录搜索方法与盲区，并迁移在覆盖范围内找到的使用端；
3. 移除旧函数、常量、类型、状态集合与行内判断；
4. 未经明确授权，不得更改公开或持久化标识符；
5. 保留并说明有意存在的差异与兼容性名称；
6. 使用现有测试框架补充测试；若项目没有测试框架，不得擅自引入；
7. 执行项目现有的测试、类型检查、lint 与 build；
8. 搜索旧名称、复制条件、magic values 与跨 repository 漂移；
9. 按完成报告格式报告结果。

若发现未授权的行为差异或规则冲突，停止并报告，不得自行选择。
```

## 完成报告格式

```text
1. 业务事实：
2. 权威名称与文件：
3. 定义数量（修改前／修改后）：
4. 搜索方法与覆盖范围：
5. 已知搜索盲区：
6. 已迁移使用端：
7. 已移除的旧定义与保留的兼容性名称：
8. 有意保留的差异：
9. 授权的行为或标识符变更：
10. 测试、类型、lint 与 build 结果：
11. clone／旧名称／magic value 检查结果：
12. 跨 repository 访问与漂移结果：
13. 尚未解决的冲突或风险：
```

没有使用某项工具或项目不存在某项检查时，应明确写“未使用”或“项目未提供”，不要假装已经执行。

## 可选安全限制

工程操作权限不属于这个 Skill 的核心规则，应由全局规则、项目规则或单次任务决定。需要限制时，可以在提示词最后加入：

```text
本次不得 commit、push、创建 PR、执行 migration 或部署。
不得修改指定范围以外的文件；若需要扩大范围，先停止并报告。
```
